# AI 协作整理说明 / Assistant Guide

这个文件是给 AI 助手看的。以后当我让你整理这个仓库里的内容时，请按照这里的规则工作。

## 仓库定位 / Repository Purpose

这个仓库是我的个人沉淀系统，用来长期整理：

- 算法刷题与思考
- 软件工程经验
- 逻辑、认知与思维方式
- 对观点、视频、评论的判断
- 文章草稿与长期输出
- 阅读、资料、灵感与复盘

它不是单纯的资料收藏夹，也不是只放代码的仓库。

核心链路：

```text
输入 → 思考 → 结构化 → 复盘 → 沉淀 → 输出
```

整理时不要只做“归档”，而要帮助我把零散内容变成可复用的判断框架。

## 总原则 / General Principles

### 1. 保留我的原始思考

不要把我的表达全部改成正式文章腔。

尤其要保留：

```text
我一开始怎么想的
为什么卡住
后来怎么理解
下次应该如何识别
```

不要只留下“标准答案”。

### 2. 结构化，而不是美化

整理的重点不是把文字写漂亮，而是让思路更清楚。

优先补充：

- 背景 / Background
- 第一反应 / First Reaction
- 问题结构 / Structure
- 边界 / Boundaries
- 反例 / Counterexamples
- 复盘 / Reflection
- 可迁移框架 / Transferable Framework

### 3. 强调边界 / Boundaries

无论是算法、工程、观点，都要写清楚：

```text
这个判断在什么条件下成立？
在什么条件下不成立？
有没有反例？
边界在哪里？
```

不要把经验写成绝对结论。

### 4. 强调触发信号 / Signals

尤其是算法笔记，不能只写题解。

要写：

```text
看到什么关键词，应该想到什么方法？
为什么想到这个方法？
为什么其他方法不合适？
```

例如：

```text
连续子数组 + 区间和 + 有负数
→ 前缀和 + HashMap
```

### 5. 必要时图文并茂

可以使用：

- 表格
- Mermaid 图
- ASCII 图
- 示例推演
- 对比表

优先使用 Markdown 能直接展示的形式。

不要为了好看而复杂化；图应该服务于理解。

## 命名规则 / Naming Rules

### 仓库名

仓库名使用英文：

```text
notes
```

### 文件夹名

文件夹使用数字 + 英文：

```text
00-inbox/
01-thinking/
02-algorithm/
03-software-engineering/
04-opinions/
05-articles/
06-reading/
99-meta/
```

### 文件名

文件名使用英文 kebab-case：

```text
thinking-framework.md
prefix-sum.md
boundary-thinking.md
viewpoint-and-argument.md
```

### 文章标题

文章内部标题使用中文 + 英文：

```md
# 边界思维 / Boundary Thinking
```

推荐策略：

```text
路径和文件名：英文
文章标题：中文 + 英文
正文：中文为主
关键概念：附英文
```

## 中英双语策略 / Bilingual Strategy

不用每句话都中英双写，否则会增加负担。

建议：

- 标题：中文 + 英文
- 小标题：中文 + 英文
- 核心概念：中文 + 英文
- 正文：中文为主
- 关键总结：可以补一句英文

例子：

```md
# 观点和论证 / Claim and Argument

## 一句话理解 / One-line Summary

观点是你主张什么，论证是你凭什么这么说。  
A claim is what you assert; an argument is how you support it.
```

## 模块整理规则 / Module Rules

### 00-inbox

临时收集区。

不要求完整，不要求分类准确。AI 整理时先判断内容应该移动到哪个模块。

### 01-thinking

沉淀底层思维方式。

重点关注：

- 观点和论证
- 逻辑判断
- 边界思维
- 相关性与因果
- 偷换概念
- 反例意识

### 02-algorithm

算法不是只记代码，而是记识别过程。

必须尽量保留：

- 题目信号
- 暴力解
- 瓶颈
- 优化思路
- 不变量
- 边界情况
- 为什么没想到
- 下次如何识别

### 03-software-engineering

沉淀真实开发经验。

重点关注：

- 需求边界
- 领域边界
- 测试
- 架构
- 交付
- 代码质量
- 技术选型
- 问题复盘

### 04-opinions

记录观点判断。

适合整理：

- 视频观点
- 评论区言论
- 金句
- 社交媒体观点
- 自己的第一反应

重点不是反驳，而是拆出逻辑结构和边界。

### 05-articles

从笔记中发展出长文。

当某个主题在多个笔记中反复出现，可以整理成文章。

### 06-reading

阅读、课程、资料记录。

不要只摘抄，要写自己的反应、不同意的地方和可迁移点。

### 99-meta

仓库规则本身。

包括 AI 整理规则、命名规则、写作规则、复盘规则和标签规则。

## 最重要的整理原则 / Most Important Rule

以后帮我整理时，请始终遵守：

```text
不只整理内容，还整理思考路径。
不只记录结论，还记录为什么。
不只写正确答案，还写当时为什么没想到。
不只分类归档，还提炼可迁移框架。
不把经验绝对化，始终写清边界。
```
