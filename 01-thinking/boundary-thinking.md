# 边界思维 / Boundary Thinking

## 一句话理解 / One-line Summary

边界决定一个判断是否成立，也决定一件事是否可完成、可交付。  
Boundaries decide whether a judgment holds and whether something can be completed and delivered.

## 核心观点 / Core Idea

很多问题不是因为能力不够，而是边界不清。

边界不清时：

- 观点会变得绝对
- 需求会无限膨胀
- 工程会难以交付
- 算法题会看不清条件
- 讨论会变成各说各话

## 常见边界 / Common Boundaries

- 条件边界：在什么条件下成立？
- 范围边界：包含什么，不包含什么？
- 责任边界：谁负责什么？
- 交付边界：做到什么程度算完成？
- 概念边界：这个词到底指什么？

## 判断模板 / Thinking Template

```text
1. 这句话/问题在讨论什么对象？
2. 它默认了哪些条件？
3. 它在哪些场景下成立？
4. 它在哪些场景下不成立？
5. 有没有反例？
6. 更精确的说法是什么？
```

## 例子 / Example

原句：

> 代码能跑就行。

更严谨的表达：

> 对于一次性脚本、低风险原型，能跑可能已经够了；但对于核心业务、多人维护、长期迭代的系统，能跑只是最低要求，还需要测试、日志、错误处理和可维护性。

## 可迁移框架 / Transferable Framework

```text
原句：A 说明 B
判断：A 不必然推出 B
可能成立的条件：在 C 场景下，A 可能是 B 的表现
反例：在 D 场景下，A 也可能来自别的原因
更严谨版本：当 C 条件成立时，A 可能倾向于 B，但不能单独证明 B
```
